I started using Google Apps Script around 2021 to develop small tools for automating and optimizing workflows. Back then, AI and RPA automation were rarely discussed. #### Previous Projects

1. **Goal — When the public-facing email inbox receives a message, forward it to the Slack work channel.**  
   **Technically, Google Apps Script does not have an event triggered when Gmail receives an email**, so we use a different approach with a scheduled trigger running every minute (or longer): check unread emails ➡️ read content ➡️ strip HTML and forward to Slack ➡️ mark as read.  
   If you want to target specific emails (by sender or subject), set up a label filter in Gmail and change "check unread emails" to "check unread emails + label" to achieve this.  
   ref: 「 [Using Google Apps Script to Forward Gmail Emails to Slack](../d414bdbdb8c9/) 」

2. **Goal — Automatically query GA traffic/Crash-free rate and send to Slack work channel.**  
   Google Apps Script already has built-in AnalyticsData / AdSense Libraries. Just import and set parameters to use them, making integration almost painless.  
   If the app wants to query crash issue details or Top 10, just add one more step: **Firebase to BigQuery**, then import BigQuery Library in Google Apps Script and run SQL queries to achieve this.  
   ref: 「 [Crashlytics + Google Analytics Automatic Query for App Crash-Free Users Rate](../793cb8f89b72/) 」、「 [Crashlytics + Big Query to Build a More Real-Time and Convenient Crash Tracking Tool](../e77b80cc6f89/) 」

3. **Goal — Automatically consolidate operational data into Google Sheets.**  
   Following the previous method, a Web App was created to display all operational data clearly on the team's TV wall; **the technical challenge at that time was that some data was internal** and inaccessible from the external network. Fortunately, the solution was to reverse the process by scheduling internal tasks to send data daily to the Google Apps Script Web App, which then processed and filled the data back into Google Sheets.  
   ref:「 [使用 Google Apps Script 實現每日數據報表 RPA 自動化](../f6713ba3fee3/) 」

4. **Goal — Build a Simple App Packaging Platform Using Existing Resources**  
   The background was that the app packaging service moved to GitHub Actions, but non-engineering teams also needed app packaging. Previously, this was handled manually; a service was needed within the company team to connect GitHub Actions to execute packaging.  
   Using Google Apps Script Web App as the platform, access was restricted to accounts within the organization. Users filled in packaging info via a form on the platform, which then triggered GitHub Actions (GitHub API) and sent packaging Slack notifications to the recipient.  
   The packaged app was uploaded to Firebase App Distribution. **However, Google Apps Script does not have a built-in library, but [after some research, it can be done directly with googleapis](../71400d408dc8/), and in the end, it was integrated smoothly.**  
   ref: “[Using Google Apps Script Web App to Connect GitHub Actions and Build a Free, Easy-to-Use Packaging Tool Platform](../4273e57e7148/)”

There are some smaller cases not listed here. **In short, the point is that "there are always more solutions than difficulties." Actually writing the code is the easiest part; the real challenge is overcoming various obstacles to connect everything together.**

#### Technical Limitations

The premise of "there are more solutions than difficulties" is that it is **technically feasible and reasonable**; in other words, "if the direction is wrong, effort is in vain." We may not need to actually write code, but we must know when to "choose" to use Google Apps Script to develop small tools.

- **User-Agent cannot be customized:** For security and abuse prevention, the Google Apps Script User-Agent cannot be set manually. (However, other header fields can be specified.)  
  **Impact:** Some rare API services require data in the User-Agent, or target websites/APIs block the Google Apps Script User-Agent. Both cases are hard limits and cannot be bypassed.  
  **Workaround:** [If you really need to use it, you can proxy through a Cloudflare Worker.](https://zhgchg.li/posts/88f0fb935120/){:target="_blank"}

- **Maximum execution time 6 minutes:** Each run can last up to 6 minutes. For large or numerous tasks, it is recommended to split them into segments.  
  Additionally, there is a total quota limit, such as triggers running up to 90 minutes per day. **(But this seems to be a soft limit; many of my scripts exceed the quota without being blocked.)**

- **Single execution is mainly synchronous and blocking:** You cannot dispatch multiple threads to run tasks concurrently within a single execution; if tasks are time-consuming or require extensive parallel processing, it is recommended to split them into multiple executions or use scheduled triggers.

- **Cold Start:** A common issue with FaaS services is that when a task hasn't been triggered for a long time, it goes to sleep, causing a longer wait time on the next trigger.  
  This problem, combined with the requirement for synchronous (blocking) execution, makes it easy for webhook deployments to be seen as failed requests by other services, leading to repeated requests, duplicate receptions, and repeated executions.  
  For example, Slack strictly requires webhook servers to respond within 3 seconds, and this issue is frequently encountered based on past experience.

- **Web App API will redirect:** For security reasons, the Content service response will redirect to a one-time `script.googleusercontent.com` URL; HTTP clients need to support follow redirects.  
  Some Webhook services will fail if they do not automatically follow 302 redirects (e.g., Jira Webhook).

- **Web App HTML:** The webpage header includes an anti-abuse notice, true RWD cannot be achieved (the page is actually an iframe within the Google framework), the URL is ugly and cannot be customized, and it cannot support full-screen PWA pages.

- **Service Limits:** Some services have usage limits, but under normal use, you usually won’t exceed them. However, if you require high-frequency checks or real-time responses, it's easy to hit those limits.

Basically, all the necessary components for development are available. In practice, you just need to connect and combine these services to achieve process automation, for example:

- Automatically Query GA Traffic/Crash-free Rate and Send to Slack Work Channel:  
  Use Analytics Data API to fetch data, PropertiesService to store Slack Bot Token, and UrlFetchApp to call Slack Send Message API.

- Automatically aggregate operational data into Google Sheets and data dashboard web:  
  Trigger scheduled function execution, use LockService to ensure only one instance runs, fetch data with Analytics Data API, get external service data via UrlFetchApp, write to Google Sheets with SpreadsheetApp, output web app interface using HtmlService, output JSON data with ContentService plus cache data with CacheService.

#### Differences from FAAS like Cloud Function / Lambda

Google Apps Script can be seen as a FAAS scripting tool centered around Google service scenarios. It has more limitations but is **currently completely free and offers seamless integration with Google services;** other FAAS services usually require payment, though some offer free tiers, and integrating with Google services typically involves complex IAM or OAuth procedures.

#### Unsuitable Scenarios

- Not part of the Google ecosystem, for example, wanting to connect Microsoft Office or OneNote…

- A hybrid of local and cloud is more suitable for n8n or AI Agent.

- Complex calculations or large data processing cannot complete within the maximum 6-minute execution time per run.

- Large-scale web scraping or ticket-snatching programs are not feasible. Google Apps Script is easily blocked or stopped by Cloudflare's anti-scraping measures.

#### Suitable Scenarios

For personal or team workflow integration, I have many scripts managing my daily routines. For example, I get daily notifications about the website traffic overview of [zhgchg.li](https://zhgchg.li/){:target="_blank"}, GitHub Repo Issues, and even update current stock prices on a Google Sheet. For teams, the app release process can integrate with Google Calendar to check calendar events, trigger the corresponding CI/CD workflows (calling GitHub Actions), then send messages to the team Slack channel and forward app review failure emails to Slack.

  On macOS, you can generate a random string with this command: `openssl rand -hex 32`

- [optional] [Regenerate the Ministry of Transportation and Central Weather Bureau API Key](https://opendata.cwa.gov.tw/userLogin){:target="_blank"} and enter it into Google Apps Script Project -> Project Settings -> Script Properties -> `CWA_API_KEY`

> ***⚠️ According to Claude Code warning:*** *All tokens exposed in the conversation may leak and should be considered compromised. For development convenience, tokens are entered directly for verification but must be regenerated and replaced after verification.*

> ***If you are familiar with GAS, you can set up the script properties yourself from the start and then instruct the AI to use them directly.***

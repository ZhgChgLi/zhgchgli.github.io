### Introduction

I started using Google Apps Script around 2021 to develop small tools for automating and optimizing workflows. Back then, AI and RPA automation were rarely discussed. The goal was simply to improve team collaboration efficiency or enhance personal comfort. All scripts were hand-coded line by line (which now seems like an old-fashioned way of programming). **But the hardest part wasn’t the coding itself; it was whether and how to integrate with other services, and how to think outside the box for integration.**

#### Previous Projects

1. **Goal — When the public-facing email inbox receives a message, forward it to the Slack work channel.**  
   **Technically, Google Apps Script does not have an event triggered when Gmail receives an email**, so we use a different approach with a scheduled trigger running every minute (or longer): check unread emails ➡️ read content ➡️ strip HTML and forward to Slack ➡️ mark as read.  
   If you only want to target specific emails (by sender or subject), set up a Label Filter in Gmail, then change the check from unread emails to unread emails with that Label to achieve this.  
   ref: 「 [Using Google Apps Script to Forward Gmail Messages to Slack](../d414bdbdb8c9/) 」

2. **Goal — Automatically query GA traffic/Crash-free rate and send it to the Slack work channel.**  
   Google Apps Script already has built-in AnalyticsData / AdSense Libraries. You just need to import and set the parameters to use them directly, making integration almost painless.  
   If the app wants to query crash issue details or the Top 10, you just need an extra step: **Firebase to BigQuery**, then import the BigQuery Library in Google Apps Script and run SQL queries to achieve this.  
   ref: 「 [Crashlytics + Google Analytics Auto Query App Crash-Free Users Rate](../793cb8f89b72/) 」、「 [Crashlytics + Big Query Build More Real-Time Convenient Crash Tracking Tool](../e77b80cc6f89/) 」

3. **Goal — Automatically consolidate operational data into Google Sheet.**  
   Following the previous approach, a Web App was created so the entire operational data could be clearly displayed on the team's TV wall; **the technical issue at that time was that some data was internal**, inaccessible from the external network. Fortunately, the solution was to switch to an internal scheduled task that sends data daily to the Google Apps Script Web App, which then processes and writes it back to Google Sheet.  
   ref:「 [Using Google Apps Script to Automate Daily Data Reports with RPA](../f6713ba3fee3/) 」

4. **Goal — Build a Simple App Packaging Platform Using Existing Resources**  
   The background was that the app packaging service moved to GitHub Actions, but non-engineering teams also needed app packaging, which was previously handled manually. A service was needed within the company team to connect to GitHub Actions for packaging.  
   Using Google Apps Script Web App as the platform, access was restricted to organization accounts only. Users filled in packaging info on a form, which triggered GitHub Actions (via GitHub API) and sent packaging notifications to Slack.  
   The packaged app was uploaded to Firebase App Distribution. **However, Google Apps Script has no built-in library, but [after some research, it can be done directly with googleapis](../71400d408dc8/), and it was smoothly integrated in the end.**  
   ref: “[Using Google Apps Script Web App to Connect GitHub Actions and Build a Free, Easy Packaging Tool Platform](../4273e57e7148/)”

There are also some smaller cases not listed here. **In short, the point is "there are always more solutions than problems." Actually writing the code is the easiest part; the real challenge is overcoming various obstacles to integrate everything together.**

#### Technical Limitations

The premise of "there are more solutions than difficulties" is that it must be **technically feasible and reasonable**; in other words, "if the direction is wrong, effort is in vain." We may not need to actually write code, but we must know when to "choose" to use Google Apps Script to develop small tools.

- **User-Agent Cannot Be Customized:** For security and abuse prevention reasons, Google Apps Script User-Agent cannot be specified by the user. (However, other header fields can be set.)  
  **Impact:** Some rare APIs require data in the User-Agent, or target websites/APIs block the Google Apps Script User-Agent. Both cases are hard limits and cannot be bypassed.  
  **Workaround:** [If you really need to use it, you can proxy through a Cloudflare Worker.](https://zhgchg.li/posts/88f0fb935120/){:target="_blank"}

- **Maximum Execution Time 6 Minutes:** Each run can last up to 6 minutes. For large or numerous tasks, it is recommended to split them into segments.  
  Additionally, there is a total quota limit, such as Triggers running up to 90 minutes per day. **(However, this seems to be a soft limit; many of my scripts exceed this limit without being blocked.)**

- **Single execution mainly uses synchronous blocking:** You cannot dispatch multiple threads to run tasks concurrently within a single execution; if tasks are time-consuming or require extensive parallel processing, it is recommended to split them into multiple executions and use scheduled triggers.

- **Cold Start:** A common issue with Faas services is that when a task hasn't been triggered for a long time, it goes to sleep, causing a longer wait time when triggered again.  
  This problem, combined with the requirement for synchronous (blocking) execution, means that if deployed as a Webhook for other services to call, it can easily be judged as a failed request, leading to repeated requests, repeated receptions, and repeated executions.  
  For example, Slack strictly requires the Webhook server to respond within 3 seconds, and this issue has often been encountered in past experience.

- **Web App API will redirect:** For security reasons, the Content service response will redirect to a one-time `script.googleusercontent.com` URL; HTTP clients need to support following redirects.  
  Some webhook services will fail if they do not automatically follow 302 redirects (e.g., Jira Webhook).

- **Web App HTML:** The webpage header includes an anti-abuse notice. True RWD is not achievable (the page is actually an iframe inside Google's framework). The URL is ugly and cannot be customized. Full-screen PWA pages are not possible.

![](/assets/35cc65327d28/1*NI80VyLt-hHAKJskJLdTmg.png)

- **Service Limits:** There are some usage limits for services, but under normal use, you usually won't hit them. However, if you need high-frequency checks or real-time responses, it's easy to reach those limits.

#### Advantages and Features

After understanding the limitations, let's look at the advantages and features.

The biggest advantage is the seamless integration with Google's native services. The Web App access can be set to the script owner, organization members, signed-in Google accounts, or anyone, **without the need to handle complex OAuth flows yourself—just one click to set up, one click to complete, and one click to start running**:

**Built-in Services:**  
**Can be used without referencing in GAS services.**

- DocumentApp → Google Docs

- SpreadsheetApp → Google Sheets

- SlidesApp → Google Slides

- FormApp → Google Forms

- GmailApp → Gmail

- CalendarApp → Google Calendar

- DriveApp → Google Drive

- SitesApp → Google Sites

- Maps → Maps / Distance / Routes

- Translate → Translation

**Advanced Services:**
**Must be enabled in the GAS services to use.**

- Sheets API

- Drive API

- Calendar API

- Gmail API

- Analytics API / Analytics Data API

- BigQuery API

- Adsense API

- YouTube Data API

- Tasks API

- Googleapis (Other/GCP)

**Tools / System Services:**

- UrlFetchApp → Calling External APIs

- PropertiesService → Key-Value Storage

- CacheService → Cache

- LockService → Concurrency Control Prevention

- Utilities → Date / Hash / Base64

- Logger → Log Output

- HtmlService → Build Web UI, support Ajax asynchronous content updates

- ContentService → Creating an API endpoint

- Trigger → Scheduled Automatic Execution

Basically, everything needed for development is already available. In practice, you just need to connect and combine these services to achieve process automation, for example:

- Automatically Query GA Traffic/Crash-free Rate and Send to Slack Work Channel:  
  Use Analytics Data API to fetch data, PropertiesService to store Slack Bot Token, and UrlFetchApp to call Slack Send Message API.

- Automatically aggregate operational data into Google Sheet and data Dashboard Web:  
  Trigger scheduled execution of functions, use LockService to ensure only I run it, fetch data with Analytics Data API, get external service data via UrlFetchApp, write to Google Sheet with SpreadsheetApp, output Web App interface using HtmlService, output JSON data with ContentService plus data caching via CacheService.

#### Differences from FAAS like Cloud Function / Lambda

Google Apps Script can be seen as a FAAS scripting tool centered around Google services. It has more limitations but is **currently completely free and seamlessly integrates with Google services;** other FAAS services usually require payment, though some offer free tiers, and integrating with Google services involves formal IAM or OAuth processes that are more complex and cumbersome.

#### Unsuitable Scenarios

- Not part of the Google ecosystem, such as wanting to connect with Microsoft Office or OneNote…

- Hybrid local and cloud setup is more suitable for n8n or AI Agent.

- Complex calculations or large data processing may not complete within the 6-minute maximum execution time per run.

- Large-scale web scraping or ticket-snatching programs are out of the question. Google Apps Script is easily blocked or stopped by Cloudflare's anti-scraping measures.

#### Suitable Scenarios

For personal or team workflow integration, I have many scripts managing my daily routines. For example, I get daily notifications of [zhgchg.li](https://zhgchg.li/){:target="_blank"} website traffic overview, GitHub Repo Issues, and even update the current stock prices in a Google Sheet. For teams, the app release process can integrate with Google Calendar to check events and trigger the corresponding CI/CD process (calling GitHub Actions), then send messages to the team Slack channel and forward app review failure emails to Slack.

3. Save clasprc (Clasp Token) to Keychain to allow AI Agent secure access

```bash
security add-generic-password \
  -U \
  -s "com.google.clasp" \
  -a "$USER" \
  -w "$(cat ~/.clasprc.json)"
```

If you want to copy the Clasp Token to put into CI/CD Secrets for running tasks, you can use `cat ~/.clasprc.json \\| base64 \\| pbcopy` (remember to add base64 encode). When using it, run `echo "$CLASPRC" \\| base64 --decode > ~/.clasprc.json && chmod 600 ~/.clasprc.json`.

#### Problem

- **Interval Overlay**  
  The same interval can have multiple overlapping states, for example, [15,29] is STRONG + EM

- **Interval Overlap**  
  STRONG: 0 ───────── 29  
  EM:□□□□□□□□□□15 ───────── 55  
  In extreme cases, users may set overlapping styles: [0,29] for STRONG and [15,55] for EM.

- **Continuous Interval Merging**  
  Data may provide [0,12] as STRONG / [12,29] as STRONG, which need to be merged into [0,29] as STRONG

Overlapping intervals and continuous merging are relatively easy to handle. The most troublesome part is managing intersecting and closing intervals. You can't render them directly by index, as it would become `**AA_AA**BBB_`. You need to handle the opening and closing yourself to get `**AA_AA_**_BBB_`, which is the correct Markdown.

iOS developers who have used `NSAttributedString attributes` face the same issue, but Apple Foundation handles the interval merging and overlay processing for us.

At that time, I solved it with the most brute-force walkthrough method, which was correct but had terrible performance and code quality.*

### Rangeable RFC

[![](https://opengraph.githubassets.com/571d1d5073b39443655e07830796836fd75a6da6537abf69ebc615e70e2b0c19/ZhgChgLi/RangeableRFC)](https://github.com/ZhgChgLi/RangeableRFC/blob/main/RFC.md){:target="_blank"}

#### TL;DR

> `Rangeable<Element>` *is a generic data structure for managing "which elements are active within integer closed intervals." It can automatically merge overlapping or adjacent intervals of the same element, supports querying which elements are active at a given index, and outputs open/close transition events within a range. It was originally designed to solve Medium Markdown markup rendering issues, such as determining which styles like `STRONG`, `EM`, or `LINK` apply to a character; however, it can also be applied to calendars, game states, genome annotation, AVPlayer byte-range caching, and more. Its core value is abstracting "interval merging, active set querying, and boundary event generation" into a deterministic, cross-language consistent specification suitable for Ruby/Swift implementations.*

> The following RFC details can be skipped unless you are very interested. This section is also directly translated and summarized by AI.

`Rangeable<Element>` is a generic data structure that **maps elements to multiple integer closed intervals and allows fast queries of which elements are active at a given position**.

It doesn't just solve a simple `Range` problem; instead, given many `(element, start, end)` tuples, I need to be able to:

1. Check which merged ranges contain a specific element

2. Check which elements are active at a certain index

3. Check which open/close boundary events exist within a certain range

The RFC clearly defines `Rangeable` as a language-neutral, generic, integer-coordinate, closed-interval set container. Elements must be hashable and compared by value equality. It supports three types of queries: `getRange`, `r[i].objs`, and `transitions`.

#### Main Motivation

This RFC originated from the Medium markup render issue in `ZMediumToMarkdown`.

Medium paragraphs contain markups like these:

```makefile
STRONG: [2, 5]
STRONG: [3, 7]
EM:     [4, 10]
```

Currently, rendering requires scanning each character to determine which tags are active, such as bold, italic, link, code, etc. The RFC mentions that the original approach converts markups into `TagChar`, sorts them by `startIndex`, and then linearly scans all tags for each character, resulting in a worst-case complexity of `O(L · m)`. `Rangeable` aims to abstract this into a general container, allowing the renderer to simply insert markups and then query using `r[i].objs` or `transitions`.

The RFC also extends this problem to other scenarios, such as:

![](/assets/f7bec8f79c08/1*AWs90nwJGkKee9iEzLwexQ.png)

The RFC clearly states the common problem: given many `(eᵢ, lᵢ, hᵢ)`, when querying a position `i`, return all elements that satisfy `l ≤ i ≤ h`.

#### Core Concept Translation

#### 1. `Rangeable<Element>`

It can be thought of as:

```css
Element -> [ClosedRange<Int>]
```

But it is not an ordinary Dictionary because it will:

1. Automatically merge overlapping intervals of the same element

2. Automatically merge adjacent ranges

3. Preserve the original insertion order of elements

4. Support fast queries for elements active at a given index

5. Support output of open/close transition events

For example:

```scss
insert(STRONG, 2, 5)
insert(STRONG, 3, 7)
```

The final result will be:

```css
STRONG -> [(2, 7)]
```

Because `[2,5]` and `[3,7]` overlap.

#### API Highlights

The main APIs defined by the RFC are as follows.

#### `insert(e, start, end)`

```sql
insert(e: Element, start: Int, end: Int)
```

The effect is:

```sql
R(e) = canonicalize(R(e) ∪ [start, end])
```

That is, add the new interval with element `e` to the existing set of intervals, then canonicalize: merge all overlapping or adjacent intervals and sort them. `start > end` must throw an `InvalidIntervalError`; inserting the same content repeatedly is idempotent, should not change the result, nor increase the version.

#### `r[i].objs` / `activeAt(index:)`

```less
r[i] -> Slot
r.activeAt(index: i) -> Slot
```

Return the active elements at a given index.

For example:

```scss
insert(Strong, 2, 5)
insert(Italic, 3, 7)
```

Query:

```css
r[3].objs
```

The result is:

```csharp
[Strong, Italic]
```

The target query complexity is:

```scss
O(log M + r)
```

Here, `M` is the total number of intervals after merging all elements, and `r` is the actual number of elements returned at that position.

#### `getRange(of:)`

```scss
getRange(of e) -> [(Int, Int)]
```

Return the canonical ranges currently merged for a given element.

For example:

```scss
insert(Strong, 2, 4)
insert(Strong, 5, 7)
```

Because `[2,4]` and `[5,7]` are adjacent on the integer coordinate, the result is:

```css
[(2, 7)]
```

The RFC explicitly requires the returned intervals to be sorted, non-overlapping, and non-adjacent.

#### `transitions(over:)`

```scss
transitions(over: ClosedRange<Int>) -> [TransitionEvent]
```

Return the open/close boundary events within a range.

For example:

```scss
insert(Strong, 2, 5)
insert(Italic, 3, 7)
```

Then the transitions will be:

```lua
[
  (2, open,  Strong),
  (3, open,  Italic),
  (6, close, Strong),
  (8, close, Italic)
]
```

Note that the close event position is `hi + 1` because the external semantics use a closed interval `[lo, hi]`, but the sweep-line internally uses "the first position no longer active" as the close coordinate. The RFC clearly states that `transitions(over: lo..hi)` will include the close event at `hi + 1`, making it easier to handle the right boundary.

#### Interval Semantics

#### 1. `end` is inclusive

This RFC is very clear: `insert(e, start: a, end: b)` represents the closed interval `[a, b]`, where `b` itself is also included in the active range.

That is to say:

```scss
insert(Strong, 2, 5)
```

Representative:

```typescript
2, 3, 4, 5 are all active
```

Not `[2, 5)`.

The reasons for choosing inclusive in the RFC include:

1. Historical Data Model Compatible with Medium Markup / ZMediumToMarkdown

2. More aligned with the human intuition of "whether this character is active"

3. Integer adjacency can be written as `hi + 1 == lo`

4. A single-point interval `[k, k]` can naturally represent a valid position.

#### 2. Adjacent intervals must be merged

On integer coordinates:

```csharp
[2, 4] + [5, 7] => [2, 7]
```

Because there is no integer position between 4 and 5 to represent "not active," the RFC requires that adjacent integer intervals of the same element must be merged.

But:

```csharp
[2, 4] + [6, 7] => [(2, 4), (6, 7)]
```

Because there is a gap of `5` in between.

#### 3. `start == end` is a valid singleton interval

```scss
insert(e, 5, 5)
```

Represents only index `5` being active. The RFC requires this to be a valid and non-empty range.

#### 4. `start > end` must throw an error

```scss
insert(e, 10, 5)
```

Cannot automatically reverse to `[5,10]`, nor silently normalize. The RFC requires throwing an `InvalidIntervalError` and the container state must remain unchanged.

#### Element equality semantics

`Rangeable` determines whether two elements are the same by using the language's native value equality.

Ruby:

```bash
eql? + hash
```

Swift:

```typescript
Hashable / Equatable
```

So:

```scss
Link("a") and Link("a") are considered the same element, so intervals will merge
Link("a") and Link("b") are different elements and will not merge
Strong() and Strong() will also merge if their equality matches
```

The RFC requires equality to satisfy reflexive, symmetric, transitive properties, and hash consistency. Additionally, once an element is inserted, external mutations should not break its hash or equality.

#### Sorting Rules

This is a very important part of the RFC.

#### Active set sorting

The order of `r[i].objs` is not based on hash, alphabetical order, or range length, but rather:

```lua
The order in which elements are first inserted
```

For example:

```scss
insert(Strong, 1, 10)
insert(Italic, 1, 10)
insert(Code, 1, 10)
```

Then:

```typescript
r[5].objs == [Strong, Italic, Code]
```

The RFC states that this approach ensures determinism, cross-language consistency, and aligns with Markdown nesting: styles that appear earlier are usually outer layers.

#### Merge does not change insertion order

For example:

```scss
insert(Strong, 1, 5)   // Strong ord = 1
insert(Italic, 3, 7)   // Italic ord = 2
insert(Strong, 4, 8)   // Strong ranges merge into [1,8], but ord remains 1
```

Query:

```css
r[6].objs
```

The result is:

```csharp
[Strong, Italic]
```

Although Strong extends to index 6 only in the third step, its first-insert order is still earlier than Italic. The RFC uses this case to fix the sorting rule as "the order of the element's first appearance."

#### Transitions Sorting

At the same coordinate:

1. `.open` comes before `.close`

2. Multiple `.open` entries: ascending by insertion order

3. Multiple `.close`: in descending insertion order, i.e., LIFO

This complies with Markdown stack discipline:

```lua
** open
_ open
_ close
** close
```

The RFC clearly defines this tie-breaking method to avoid inconsistent Ruby / Swift hash orders causing different outputs across languages.

#### Internal Data Structure

The core design chosen for the RFC is:

```
Map<Element, SortedList<Interval>>
+ insertion_order
+ lazy boundary-event index
+ version counter
```

#### 1. Per-element sorted disjoint list

Each element has its own interval list:

```swift
intervals: Map<Element, SortedList<Interval>>
```

And it must maintain the canonical form:

1. Sort by `lo` in ascending order

2. Non-overlapping

3. Non-adjacent to each other

4. Each interval satisfies `lo <= hi`

The RFC calls this the I1 invariant.

#### 2. Lazy boundary-event index

`Rangeable` does not rebuild the query index on every insert; it only builds it at the first query.

```makefile
version: Int
event_index: EventIndex?
```

`event_index` includes:

```vbnet
events: SortedArray<Event>
segments: SortedArray<Segment>
version: Int
```

This design is meant to fit the "build-once-then-query-densely" workload: after a large number of inserts, queries at each position are performed intensively. The RFC clearly states that a lazy index suits this pattern because the build phase does not require constant index rebuilding, and the query phase incurs a one-time cost of `O(M log M)`.

#### Algorithm Highlights

#### Core process of `insert`

The RFC pseudocode is roughly:

```bash
function insert(r, e, start, end):
  if start > end:
    raise InvalidIntervalError

  e_frozen = freeze_for_insert(e)

  if e not in intervals:
    intervals[e] = []
    insertion_order.append(e)
    ord[e] = insertion_order.length

  list = intervals[e]
  lo = start
  hi = end

  Find the first interval that may overlap or be adjacent to [lo, hi]

  while interval overlaps or is adjacent to [lo, hi]:
    lo = min(lo, interval.lo)
    hi = max(hi, interval.hi)
    Remove the old interval

  Insert the merged new [lo, hi]

  If there is a real change:
    version += 1
    event_index = nil
```

Pay special attention to the RFC's warning not to use `lo - 1` for checks, as it will underflow when `lo == Int.min`; instead, use `hi + 1 >= lo` or an equivalent successor model.

#### Complexity

The reference structure chosen for the RFC is:

```csharp
per-element sorted disjoint list + lazy event index
```

The complexity is roughly as follows:

![](/assets/f7bec8f79c08/1*5HKeVhHLMRIWnuJJmmGTbQ.png)

Among them:

```ini
m_e = The number of merged intervals for a certain element itself
k = The number of old intervals absorbed/merged in this insert
M = The total number of merged intervals for all elements
r = The number of elements actually active at the query position
```

The RFC also points out that if there are indeed `m` active elements at the same position, the output itself requires `Ω(m)`, so `O(log M + m)` is already a reasonable output-sensitive bound.

#### Why Not Use Interval Tree / Segment Tree / Roaring Bitmap?

The RFC has a whole section comparing alternative solutions, ultimately choosing `(a) Per-element sorted disjoint list + lazy event index`.

Key points are as follows:

![](/assets/f7bec8f79c08/1*psrNFxHTKrwba92dA0rOLw.png)

Three reasons for choosing the main structure in the RFC summary:

1. The meaning of per-element merge is natural, and `getRange` can directly return `R(e)`

2. Deterministic and cross-language reproducible

3. Lazy index is well suited for build-once-then-query workloads

#### Features not included in v1

RFC explicitly lists what v1 will not do:

```sql
remove(e, start, end)
remove(e)
clear
union / intersect / difference
persistent immutable snapshot
floating-point coordinates
multi-dimensional rectangle stabbing
r[lo...hi].objs range slot query
```

### Admiring AI

I originally only used AI for application problems, such as [redesigning a website](../6bf79c5b4dab/), [personal dashboard](../35cc65327d28/), or product issue fixes at work; this time, I tried having it deeply study and solve algorithm and underlying data design problems for the first time.

The results exceeded my expectations. After roughly reading the RFC it wrote, both the format and content are as good as those researched and written by real people (at least definitely better than what I could write). I observed what it was doing: it uses Web Search to find publicly available related papers or technical documents, then integrates and evaluates the feasibility of the implementation. Splitting the roles into doer and reviewer is also very effective—the doer focuses too much on implementation details, while the reviewer takes a broader and higher-level view to check if the doer's work has any side effects. Finally, after the RFC is finalized, I ask the AI to implement based on this document, and the accuracy is almost 100%.

#### Reflections on AI Usage

But there's no need to fear being replaced by AI; the key is problem-solving thinking, which AI is still weak at. For example, if you ask it to optimize ZMediumToMarkdown directly, it might only focus on the Medium scenario and the original code style. **However, humans know to abstract it into a Foundation, study the RFC first, and then implement it, which works much better. Of course, we can do it ourselves, but it takes time. AI helps speed up this process, not replace us.**

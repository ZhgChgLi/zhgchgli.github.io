#### 問題

- **区間の重ね合わせ**  
  同じ区間に複数の状態が重なることがあります。例えば、[15,29] は STRONG と EM が重なっています。

- **区間の重なり**  
  STRONG: 0 ───────── 29  
  EM:□□□□□□□□□□15 ───────── 55  
  極端な場合、ユーザーは重なったスタイルを設定することがあります。[0,29] は STRONG、[15,55] は EM です。

- **区間の連続マージ**  
  データで [0,12] が STRONG、[12,29] が STRONG の場合、[0,29] にマージする必要があります。

区間の重なりや連続した区間の結合は比較的簡単ですが、一番厄介なのは区間の交差閉合の処理です。インデックス通りにレンダリングすると `**AA_AA**BBB_` のようになり、正しくありません。開閉を自分で処理して `**AA_AA_**_BBB_` にしなければ、正しいMarkdownになりません。

iOS 開発者で `NSAttributedString attributes` を使ったことがある人も同じ問題に直面しますが、Apple Foundation は区間の結合や重複処理をすでに行ってくれています。

> *当時もかなり時間をかけて考えましたが、実際に使う時になって知識が足りないことを痛感しました。普段あまり問題を解いていなかったため、実用時に武器がありませんでした。その時は最も原始的なウォークスルーで解決しましたが、正確ではあっても性能とコードがひどいものでした。*

以上の図を例に、**現在のデータ区間：** `[250,566] [850,959]` とすると、理想的には：

### Rangeable RFC

[![](https://opengraph.githubassets.com/571d1d5073b39443655e07830796836fd75a6da6537abf69ebc615e70e2b0c19/ZhgChgLi/RangeableRFC)](https://github.com/ZhgChgLi/RangeableRFC/blob/main/RFC.md){:target="_blank"}

#### TL;DR

> `Rangeable<Element>` *は「要素がどの整数の閉区間内で有効か」を管理する汎用データ構造です。同じ要素の重複または隣接する区間を自動的に統合し、特定のインデックスでどの要素がアクティブかを検索したり、ある範囲内の開閉トランジションイベントを出力したりできます。もともとは Medium の Markdown マークアップレンダリング問題、例えばある文字が同時に `STRONG`、`EM`、`LINK` のどのスタイルを適用されているかを判定するために作られましたが、**カレンダー、ゲーム状態、ゲノム注釈、AVPlayer のバイトレンジキャッシュなどの場面にも適用可能です。** コアバリューは「区間の統合、アクティブセットの検索、境界イベントの生成」を決定論的かつ言語間で一貫した仕様として抽象化し、Ruby / Swift での実装に適した設計にしたことです。*

> 以下のRFCの詳細は、特に興味がなければそのまま読み飛ばして問題ありません。ここもAIによる翻訳の要約を使用しています。

`Rangeable<Element>` は「**要素を複数の整数閉区間に対応させ、特定の位置でどの要素が有効かを高速に検索できる**」汎用データ構造です。

それは単なる `Range` の問題を解決するのではなく、多くの `(要素, start, end)` を与えられたときに、次のことができるようにする必要があります：

1. ある要素が属する結合後の区間を調べる

2. あるインデックスでどの要素がアクティブかを調べる

3. ある範囲内にどのような open / close 境界イベントがあるか確認する

RFC は `Rangeable` を言語に依存しない、ジェネリックな整数座標の閉区間集合コンテナとして明確に定義しています。要素はハッシュ可能で、値の等価性によって比較されます。`getRange` 、 `r[i].objs` 、 `transitions` の3種類のクエリをサポートします。

#### 主な動機

この RFC は `ZMediumToMarkdown` の Medium マークアップレンダリングの問題に起因しています。

Medium の段落内には、このようなマークアップがあります：

```makefile
STRONG: [2, 5]
STRONG: [3, 7]
EM:     [4, 10]
```

現在のレンダリングでは、各文字ごとにどのタグがアクティブか（例えば bold、italic、link、code など）を逐一スキャンして判定しています。RFC には、従来の方法では markups を `TagChar` に変換し、`startIndex` に基づいてソートした後、各文字で全てのタグを線形にスキャンするため、最悪で `O(L · m)` になると記載されています。`Rangeable` はこの処理を汎用コンテナとして抽象化し、レンダラーは単に markups を挿入し、`r[i].objs` や `transitions` で問い合わせるだけで済むようにしたいと考えています。

RFC はこの問題を他のシナリオにも拡張しています。例えば：

![](/assets/f7bec8f79c08/1*AWs90nwJGkKee9iEzLwexQ.png)

RFC は共通の問題として明確に指摘しています：多くの `(eᵢ, lᵢ, hᵢ)` が与えられたとき、ある位置 `i` を検索すると、`l ≤ i ≤ h` を満たすすべての要素を返す必要がある。

#### コアコンセプト翻訳

#### 1. `Rangeable<Element>`

こう考えることができます：

```css
Element -> [ClosedRange<Int>]
```

しかし、それは普通の Dictionary ではありません。なぜなら、次のことを行うからです：

1. 同じ要素の重複区間を自動的にマージする

2. 隣接する区間を自動的にマージする

3. 要素の初回挿入順序を保持する

4. 特定のインデックスでアクティブな要素の高速検索をサポートする

5. オープン / クローズのトランジションイベントの出力をサポートする

例えば：

```scss
insert(STRONG, 2, 5)
insert(STRONG, 3, 7)
```

最後は次のようになります：

```css
STRONG -> [(2, 7)]
```

なぜなら `[2,5]` と `[3,7]` が重なっているからです。

#### API のポイント

RFCで定義された主なAPIは以下の通りです。

#### `insert(e, start, end)`

```sql
insert(e: Element, start: Int, end: Int)
```

効果は：

```sql
R(e) = canonicalize(R(e) ∪ [start, end])
```

つまり、新しい区間を要素 `e` の既存の区間集合に追加し、canonicalize を行う：重複または隣接するすべての区間をマージしてソートする。`start > end` の場合は `InvalidIntervalError` を投げる必要がある；同じ内容を繰り返し挿入しても冪等であり、結果を変更せず、version も増加しない。

#### `r[i].objs` / `activeAt(index:)`

```less
r[i] -> スロット
r.activeAt(index: i) -> スロット
```

ある index 上でアクティブな要素を返す。

例えば：

```scss
insert(Strong, 2, 5)
insert(Italic, 3, 7)
```

検索：

```css
r[3].objs
```

結果は：

```csharp
[Strong, Italic]
```

クエリの複雑度の目標は：

```scss
O(log M + r)
```

ここで `M` はすべての要素をマージした後の interval の総数で、`r` はその位置で実際に返される要素の数です。

#### `getRange(of:)`

```scss
getRange(of e) -> [(Int, Int)]
```

ある要素の現在の統合後の canonical ranges を返します。

例えば：

```scss
insert(Strong, 2, 4)
insert(Strong, 5, 7)
```

なぜなら `[2,4]` と `[5,7]` は整数座標上で隣接しているため、結果は次の通りです：

```css
[(2, 7)]
```

RFC は、返される intervals が必ずソートされており、重複や隣接がないことを明確に要求しています。

#### `transitions(over:)`

```scss
transitions(over: ClosedRange<Int>) -> [TransitionEvent]
```

指定範囲内の open / close 境界イベントを返します。

例えば：

```scss
insert(Strong, 2, 5)
insert(Italic, 3, 7)
```

すると transitions は以下のようになります：

```lua
[
  (2, open,  Strong),
  (3, open,  Italic),
  (6, close, Strong),
  (8, close, Italic)
]
```

注意すべきは close event の位置が `hi + 1` であることです。外部の意味としては閉区間 `[lo, hi]` ですが、sweep-line 内部では「最初に active でなくなる位置」を close coordinate としています。RFC では `transitions(over: lo..hi)` が `hi + 1` の close event を含むことを明確に説明しており、右端の処理を容易にしています。

#### 区間の意味

#### 1. `end` は包含される

このRFCは非常に明確です：`insert(e, start: a, end: b)` は閉区間 `[a, b]` を表し、`b` 自身もアクティブレンジに含まれます。

つまり：

```scss
insert(Strong, 2, 5)
```

代表：

```typescript
2, 3, 4, 5 はすべてアクティブ
```

`[2, 5)` ではありません。

RFCでinclusiveを選んだ理由は以下の通りです：

1. Medium マークアップ / ZMediumToMarkdown に対応した履歴データモデル

2. 「この文字がアクティブかどうか」という人間の直感に適しています

3. 整数の隣接結合は `hi + 1 == lo` と表現できます

4. 単一区間 `[k, k]` は自然に有効な位置を表せます

#### 2. 隣接する区間は結合する

整数座標上で：

```csharp
[2, 4] + [5, 7] => [2, 7]
```

4 と 5 の間に「非アクティブ」を表す整数位置がないため、RFCでは同じ要素の隣接する整数区間を必ず結合する必要があります。

しかし：

```csharp
[2, 4] + [6, 7] => [(2, 4), (6, 7)]
```

なぜなら間に `5` というギャップがあるからです。

#### 3. `start == end` は有効なシングルトン区間です

```scss
insert(e, 5, 5)
```

代表インデックス `5` のみがアクティブです。RFCでは、これは有効かつ空でない区間である必要があります。

#### 4. `start > end` は必ずエラーを投げる必要がある

```scss
insert(e, 10, 5)
```

自動的に `[5,10]` に反転することも、サイレントに正規化することもできません。RFCでは必ず `InvalidIntervalError` をスローし、かつコンテナの状態を変更してはいけません。

#### Element equality の意味

`Rangeable` は、2つの要素が同じ要素かどうかを判断する際に、言語固有の値の等価性を使用します。

Ruby：

```bash
eql? + hash
```

Swift：

```typescript
Hashable / Equatable
```

だから：

```scss
Link("a") と Link("a") は同じ要素とみなされ、区間は結合されます
Link("a") と Link("b") は異なる要素であり、結合されません
Strong() と Strong() は equality が等しければ、結合されます
```

RFC は、等価性が反射律、対称律、推移律、ハッシュの一貫性を満たす必要があり、要素が挿入された後は外部からの変更によってハッシュや等価性が破壊されないことを要求しています。

#### ソートルール

これは RFC の非常に重要な部分です。

#### アクティブセットのソート

`r[i].objs` の順序はハッシュでもアルファベット順でもレンジの長さでもなく、以下の通りです：

```lua
要素が初めて挿入された順序
```

例えば：

```scss
insert(Strong, 1, 10)
insert(Italic, 1, 10)
insert(Code, 1, 10)
```

則：

```typescript
r[5].objs == [Strong, Italic, Code]
```

RFC は、これを行う理由として、決定論的で言語間の一貫性があり、Markdown のネストに適合していることを指摘しています：先に出現するスタイルは通常外側になります。

#### Merge は挿入順序を変更しません

例えば：

```scss
insert(Strong, 1, 5)   // Strong の順序 = 1
insert(Italic, 3, 7)   // Italic の順序 = 2
insert(Strong, 4, 8)   // Strong の範囲は [1,8] にマージされるが、順序は 1 のまま
```

検索：

```css
r[6].objs
```

結果は：

```csharp
[Strong, Italic]
```

Strong はステップ3で index 6 に拡張されましたが、その first-insert order は Italic よりも早いです。RFC はこのケースを使って「要素が初めて現れた順序」がソート基準であることを明確にしています。

#### Transitions の並び替え

同じ座標上で：

1. `.open` は `.close` よりも先に来る

2. 複数の `.open` は挿入順に昇順で並ぶ

3. 複数の `.close` は挿入順の降順、つまりLIFOとなる

このようにすると Markdown のスタック規律に適合します：

```lua
** 開く
_ 開く
_ 閉じる
** 閉じる
```

RFC は、このタイブレークルールを明確に定義し、Ruby と Swift のハッシュ順序の不一致によるクロス言語の出力差異を防ぎます。

#### 内部データ構造

RFCで選択したコア設計は：

```
Map<Element, SortedList<Interval>>
+ 挿入順序
+ 遅延境界イベントインデックス
+ バージョンカウンター
```

#### 1. 要素ごとのソート済み非重複リスト

各要素はそれぞれ自分の interval list を持っています：

```swift
intervals: Map<Element, SortedList<Interval>>
```

そして、canonical form を維持しなければなりません：

1. `lo` を昇順にソートする

2. 重ならない

3. 隣接していない

4. 各区間は常に `lo <= hi` を満たす。

RFC ではこれを I1 不変条件と呼びます。

#### 2. 遅延境界イベントインデックス

`Rangeable` は毎回 insert のたびに検索インデックスを再構築するのではなく、最初のクエリ時に初めて構築します。

```makefile
version: Int
event_index: EventIndex?
```

`event_index` には以下が含まれます：

```vbnet
events: SortedArray<Event>
segments: SortedArray<Segment>
version: Int
```

この設計は「build-once-then-query-densely」のワークロードに対応するためのものです：大量の挿入(insert)を終えた後、各位置を密にクエリします。RFCでは、lazy indexがこのパターンに適していると明確に説明しています。なぜなら、構築フェーズではインデックスを何度も再構築する必要がなく、クエリフェーズでは一度だけ `O(M log M)` のコストを払えば良いためです。

#### アルゴリズムのポイント

#### `insert` のコアプロセス

RFC の擬似コードは大まかに以下の通りです：

```bash
function insert(r, e, start, end):
  if start > end:
    raise InvalidIntervalError

  e_frozen = freeze_for_insert(e)

  if e not in intervals:
    intervals[e] = []
    insertion_order.append(e)
    ord[e] = insertion_order.length

  list = intervals[e]
  lo = start
  hi = end

  # [lo, hi] と重複または隣接する最初の interval を見つける

  while interval が [lo, hi] と重複または隣接している:
    lo = min(lo, interval.lo)
    hi = max(hi, interval.hi)
    # 古い interval を削除する

  # マージ後の新しい [lo, hi] を挿入する

  if 変更があった場合:
    version += 1
    event_index = nil
```

特に RFC では、`lo - 1` を使って判定してはいけないと注意しています。`lo == Int.min` の場合にアンダーフローが発生するためです。代わりに `hi + 1 >= lo` または同等の successor モデルを使うべきです。

#### 複雑度

RFC で選んだ reference structure は：

```csharp
要素ごとにソートされた非重複リスト + 遅延イベントインデックス
```

複雑度はおおよそ以下の通りです：

![](/assets/f7bec8f79c08/1*5HKeVhHLMRIWnuJJmmGTbQ.png)

その中で：

```ini
m_e = ある element 自身のマージされた区間の数
k = 今回の insert で吸収・合併された古い区間の数
M = すべての elements のマージされた区間の合計数
r = クエリ位置で実際にアクティブな elements の数
```

RFC はまた、同じ位置に実際に `m` 個の要素がアクティブであれば、出力自体が `Ω(m)` を必要とするため、`O(log M + m)` はすでに出力に依存した妥当な境界であると指摘しています。

#### なぜ Interval Tree / Segment Tree / Roaring Bitmap を使わないのか？

RFC には代替案のセクションがあり、最終的に `(a) Per-element sorted disjoint list + lazy event index` を選択しました。

ポイントは以下の通りです：

![](/assets/f7bec8f79c08/1*psrNFxHTKrwba92dA0rOLw.png)

RFC の要約：主な構造を選んだ3つの理由：

1. per-element merge の意味は自然で、`getRange` は直接 `R(e)` を返すことができます

2. 決定論的で、クロス言語間で再現可能です

3. lazy index は一度構築してからクエリを行うワークロードに非常に適しています

#### v1 に含まれていない機能

RFC は v1 で行わないことを明確に列挙しています：

```sql
remove(e, start, end)
remove(e)
clear
union / intersect / difference
persistent immutable snapshot
floating-point coordinates
multi-dimensional rectangle stabbing
r[lo...hi].objs のような range slot query
```

削除や交差の部分もAIにRFC V2の実装を任せました。
